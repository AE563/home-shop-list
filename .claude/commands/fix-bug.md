# /fix-bug — TDD-цикл для исправления бага

**Правило:** тест пишется ДО фикса. Не наоборот.

---

## Фаза 1 — Понять баг

Прочитай описание бага. Найди в коде место, где происходит:
- Какое поведение ожидается?
- Какое фактическое?
- В каком файле/функции корень проблемы?

Не трогай код.

---

## Фаза 2 — Написать воспроизводящий тест (RED)

Напиши тест, который:
- Воспроизводит баг — **проверяет ожидаемое поведение**, которого сейчас нет
- Максимально изолирован (unit), если возможно
- Расположен в правильном файле (`test_api.py`, `test_models.py`, `ui.test.js`, `websocket.test.js`)

**Не запускай тест ещё.** Сначала напиши — потом проверь.

---

## Фаза 3 — Убедиться что тест падает

```bash
# Python
.venv2/bin/pytest --no-cov -q -k "имя_теста"

# JavaScript
npm test -- --testNamePattern="имя теста"
```

**Тест ОБЯЗАН упасть.** Если он зелёный — тест неправильно написан: он не воспроизводит баг.
Вернись на Фазу 2.

---

## Фаза 4 — Исправить код

Минимальное изменение, которое исправляет баг.
Не рефакторить попутно, не «улучшать» соседний код — только фикс.

---

## Фаза 5 — Убедиться что тест проходит

```bash
# Python
.venv2/bin/pytest --no-cov -q -k "имя_теста"

# JavaScript
npm test -- --testNamePattern="имя теста"
```

Тест ОБЯЗАН стать зелёным. Если нет — вернись на Фазу 4.

---

## Фаза 6 — Прогнать все тесты (регрессия)

```bash
.venv2/bin/pytest --no-cov -q
npm test
```

Все тесты должны быть зелёными. Если что-то сломалось — исправь.

---

## Итог

- Тест написан ДО фикса ✓
- Тест был красным (Фаза 3) ✓
- Тест стал зелёным (Фаза 5) ✓
- Регрессия чистая (Фаза 6) ✓

После — при необходимости `/check` или `пушим`.
