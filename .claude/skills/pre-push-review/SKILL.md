# Pre-Push Review

Полный пайплайн перед `git push`. Порядок фиксированный.

---

## Фаза 0 — Получить diff

```bash
git diff HEAD
```

Запомни что изменилось — агенты будут это анализировать.
Если изменений нет (`git diff HEAD` пустой) — спроси пользователя что именно ревьюить.

---

## Фаза 1 — Линтеры с авто-фиксом

Запускай **параллельно**:

```bash
# Python
.venv2/bin/ruff check apps/ tests/ --fix
.venv2/bin/ruff format apps/ tests/

# JavaScript
npx eslint static/js/ tests/js/ --fix
```

После — покажи `git diff` пользователю: что изменил линтер.

- Если после `--fix` остались **ошибки** (не warnings) — исправь вручную и повтори
- Линтер создаёт новые изменения → получи обновлённый `git diff HEAD` для агентов

---

## Фаза 2 — Три агента код-ревью (параллельно)

Запусти **все три агента одним сообщением** через Agent tool. Передай каждому полный `git diff HEAD`.

### Агент 1: Code Reuse

Для каждого изменения:
1. Поищи существующие утилиты, которые дублирует новый код (Grep по codebase)
2. Укажи новые функции, которые уже есть где-то в проекте
3. Укажи инлайн-логику, которую можно заменить существующим хелпером

### Агент 2: Code Quality

Смотри на:
1. **Дублирование состояния** — state, который можно вывести из другого state
2. **Разрастание параметров** — новые параметры вместо переработки структуры
3. **Copy-paste с вариацией** — похожие блоки, которые надо объединить
4. **Утечки абстракций** — внутренние детали, которые не должны быть публичными
5. **Строковая типизация** — raw strings вместо констант/enum

### Агент 3: Efficiency

Смотри на:
1. **Лишняя работа** — дублирующие запросы, N+1, повторные вычисления
2. **Пропущенный параллелизм** — независимые операции запущены последовательно
3. **Bloat в hot path** — блокирующая работа в startup или per-request
4. **Утечки памяти** — неограниченные структуры, потерянные event listeners
5. **Избыточные проверки существования** — проверяй через try/except, не через if exists

---

## Фаза 3 — Исправить найденное

Дождись всех трёх агентов. Агрегируй находки:

- Исправь каждую реальную проблему напрямую в коде
- Ложные срабатывания — отметь и пропусти (не спорь с агентом, просто пропусти)
- После исправлений покажи пользователю краткий список: что исправлено / что пропущено и почему

---

## Фаза 4 — Тесты

```bash
.venv2/bin/pytest --no-cov -q
npm test
```

- Все тесты должны быть зелёными
- Если красный — исправь код **или** тест, потом повтори
- Нельзя пушить с падающими тестами

---

## Фаза 5 — Django system check

```bash
.venv2/bin/python manage.py check
```

- Ожидаемый результат: `System check identified no issues (0 silenced)`
- Errors блокируют пуш, warnings — нет (но покажи пользователю)

---

## Фаза 6 — Коммит и пуш

Только если фазы 1–5 завершились без ошибок:

```bash
git status
git add <только релевантные файлы>   # НЕ git add -A
git diff --cached                     # показать пользователю staged diff
git commit -m "..."
git push
```

Если изменения из линтера или агентов значительные — создай **отдельный коммит**:
```
style: auto-fix ruff/eslint
```
и затем основной коммит с фичей.

### Conventional Commits

```
feat:     новая функциональность
fix:      исправление бага
refactor: рефакторинг без изменения поведения
chore:    зависимости, конфиги, скрипты
test:     тесты
docs:     документация
style:    только форматирование (lint fix)
```

---

## Итоговый отчёт

По завершении выдай одну строку:

```
✅ Все проверки прошли — запушено (commit: <hash>)
```

или

```
❌ Стоп: [линтер / агент-ревью / тесты / django check] — [что именно]
```
