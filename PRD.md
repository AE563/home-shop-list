## 1. Обзор проекта

### 1.1 Краткое описание

Мобильное веб-приложение для ведения общего списка покупок для всей семьи. Хранит переиспользуемый справочник товаров по категориям и показывает, что нужно купить, в режиме реального времени.

### 1.2 Проблема / Мотивация

Каждый раз приходится заново вписывать одни и те же товары. Позиции расположены хаотично, неудобно отслеживать, что уже взяли, а что ещё осталось. При этом несколько членов семьи должны видеть актуальный список одновременно.

### 1.3 Цель MVP

Любой авторизованный член семьи может открыть приложение на смартфоне, увидеть актуальный список нужных покупок, отмечать товары по ходу похода в магазин -- и изменения мгновенно отображаются у всех остальных без перезагрузки страницы.

## 2. Границы проекта

### В рамках MVP (In Scope)

- [ ] Авторизация пользователя (email + пароль). Пользователи создаются только суперюзером -- страницы регистрации нет.
- [ ] Создание, редактирование и удаление категорий товаров с ручным приоритетом.
- [ ] Создание, редактирование и удаление товаров внутри категории (название, количество, единица измерения).
- [ ] Переключение статуса товара «нужно купить» (чекбокс).
- [ ] Страница просмотра: показывает только товары, отмеченные к покупке.
- [ ] Страница редактирования: показывает полный справочник товаров и категорий.
- [ ] Сворачивание / разворачивание категорий (accordion) на обеих страницах.
- [x] WebSocket-синхронизация: изменения одного пользователя мгновенно видны всем остальным.
- [ ] Только локальный запуск (деплой в MVP не требуется).

### За рамками MVP (Out of Scope)

- Нативное мобильное приложение (iOS / Android)
- Экспорт в PDF или Excel
- Деплой на внешний сервер
- Страница регистрации (новых пользователей создаёт суперюзер через admin или CLI)
- Управление составом семьи / приглашения пользователей
- История покупок и статистика
- Тёмная тема

### Возможно в будущем (Future Scope)

- Push-уведомления об обновлении списка покупок
- Приглашение пользователей по ссылке
- Автодеплой через GitHub Actions на VPS

## 3. Пользователи и сценарии

### 3.1 Целевой пользователь

Члены одной семьи (2--5 человек). Основной сценарий: один из членов семьи идёт в магазин со смартфоном, открывает страницу просмотра и работает со списком, отмечая товары по ходу. Другой член семьи может в это время добавить товар -- и он тут же появится на экране первого.

### 3.2 Пользовательские истории (User Stories)

> Формат: Как [роль], я хочу [действие], чтобы [ценность].
> Приоритет: [MUST] = обязательно в MVP | [SHOULD] = важно, не блокирует | [NICE] = желательно

| #     | История                                                                                                                                              | Приоритет |
|-------|------------------------------------------------------------------------------------------------------------------------------------------------------|-----------|
| US-01 | Как пользователь, я хочу войти по email и паролю, чтобы список был защищён от посторонних. Аккаунт создаёт администратор.                           | [MUST]    |
| US-02 | Как пользователь, я хочу видеть только товары, которые ещё нужно купить, чтобы не отвлекаться на уже купленные.                                     | [MUST]    |

| US-04 | Как пользователь, я хочу видеть изменения других членов семьи без перезагрузки страницы, чтобы всегда иметь актуальные данные.                      | [MUST]    |

## 4. Функциональные требования

### 4.1 Модуль: Авторизация

> Страницы регистрации нет. Аккаунты создаются администратором через
> `python manage.py createsuperuser` или Django Admin (`/admin`).

**FR-01: Вход в систему**
- **Given:** Пользователь открывает приложение и не авторизован.
- **When:** Он вводит email и пароль и нажимает «Войти».
- **Then:** При корректных данных -- редирект на страницу просмотра (`/`). При некорректных -- сообщение об ошибке: «Неверный email или пароль».

**FR-02: Выход из системы**
- **Given:** Пользователь авторизован.
- **When:** Он нажимает «Выйти» в шапке.
- **Then:** Сессия завершается. Пользователь перенаправляется на страницу входа.

**FR-03: Защита маршрутов**
- **Given:** Пользователь не авторизован.
- **When:** Он пытается открыть любую страницу кроме `/login`.
- **Then:** Автоматический редирект на `/login`.

---

### 4.2 Модуль: Категории

**FR-04: Создание категории**
- **Given:** Пользователь находится на странице редактирования.
- **When:** Он нажимает «Добавить категорию», вводит название и числовой приоритет, нажимает кнопку сохранения.
- **Then:** Категория появляется в списке на позиции согласно приоритету. При совпадении приоритета с существующей -- новая занимает эту позицию, все последующие сдвигаются на 1 вниз (каскадный сдвиг).

**FR-05: Отмена создания категории**
- **Given:** Форма создания категории открыта.
- **When:** Пользователь нажимает кнопку отмены.
- **Then:** Форма закрывается. Категория не создаётся.

**FR-06: Редактирование категории**
- **Given:** Пользователь находится на странице редактирования.
- **When:** Он редактирует название или приоритет существующей категории и сохраняет.
- **Then:** Данные обновляются. При изменении приоритета -- категория перемещается на новую позицию с каскадным сдвигом остальных.

**FR-07: Удаление категории**
- **Given:** Пользователь находится на странице редактирования.
- **When:** Он нажимает «Удалить» у категории.
- **Then:** Появляется диалог подтверждения: «Удалить категорию и все товары в ней?». При подтверждении -- категория и все её товары удаляются, список перестраивается.

---

### 4.3 Модуль: Товары

**FR-08: Создание товара**
- **Given:** Пользователь на странице редактирования, внутри существующей категории.
- **When:** Он нажимает «Добавить товар», вводит название, выбирает единицу измерения из списка, указывает количество и нажимает кнопку сохранения.
- **Then:** Товар добавляется в категорию и сортируется по алфавиту среди остальных товаров. `is_need_to_buy` по умолчанию = `True`.

**FR-09: Отмена создания товара**
- **Given:** Форма создания товара открыта.
- **When:** Пользователь нажимает кнопку отмены.
- **Then:** Форма закрывается. Товар не создаётся.

**FR-10: Редактирование товара**
- **Given:** Пользователь находится на странице редактирования.
- **When:** Он редактирует поля существующего товара и сохраняет.
- **Then:** Данные обновляются. При изменении названия -- позиция товара пересчитывается по алфавиту.

**FR-11: Удаление товара**
- **Given:** Пользователь находится на странице редактирования.
- **When:** Он нажимает «Удалить» у товара.
- **Then:** Товар удаляется без диалога подтверждения. Оставшиеся товары сохраняют алфавитный порядок.

---

### 4.4 Модуль: Просмотр и навигация

**FR-12: Страница просмотра -- отображение данных**
- **Given:** Пользователь открывает страницу просмотра (`/`).
- **When:** Страница загружается.
- **Then:** Показываются только категории, у которых есть хотя бы один товар с `is_need_to_buy = True`. Внутри -- только такие товары, отсортированные по алфавиту. Категории отсортированы по приоритету.

**FR-13: Страница редактирования -- отображение данных**
- **Given:** Пользователь открывает страницу редактирования (`/edit`).
- **When:** Страница загружается.
- **Then:** Показываются все категории и все их товары. Товары с `is_need_to_buy = False` визуально приглушены (opacity: 0.4). Сортировка: категории по приоритету, товары внутри -- по алфавиту.

**FR-14: Переключение между страницами**
- **Given:** Пользователь находится на любой странице.
- **When:** Он нажимает переключатель «Просмотр / Редактирование» в шапке.
- **Then:** Открывается противоположная страница. Текущая страница подсвечена активной в переключателе.

**FR-15: Чекбокс «нужно купить»**
- **Given:** Пользователь находится на странице просмотра.
- **When:** Он снимает галочку с товара.
- **Then:** `is_need_to_buy` устанавливается в `False`. Товар исчезает со страницы просмотра. Если это был последний видимый товар в категории -- категория тоже исчезает.

**FR-16: Accordion -- свернуть / развернуть категорию**
- **Given:** Пользователь находится на любой странице.
- **When:** Он нажимает на заголовок категории.
- **Then:** Список товаров внутри скрывается / показывается. Заголовок категории всегда виден. Состояние свёрнутости не сохраняется между сессиями.

**FR-17: Кнопка «Свернуть всё / Развернуть всё»**
- **Given:** Пользователь находится на любой странице.
- **When:** Он нажимает соответствующую кнопку в шапке.
- **Then:** Все категории одновременно сворачиваются или разворачиваются. Подпись кнопки меняется на противоположную.

---

### 4.5 Модуль: Синхронизация в реальном времени

**FR-18: WebSocket-трансляция изменений**
- **Given:** Два или более пользователей одновременно открыли приложение.
- **When:** Один из них выполняет любое действие (создание / редактирование / удаление товара или категории, переключение чекбокса).
- **Then:** Все остальные подключённые пользователи видят обновление мгновенно без перезагрузки. Собственные изменения пользователя применяются локально и не нуждаются в обратной трансляции.

**FR-19: Переподключение при разрыве соединения**
- **Given:** WebSocket-соединение пользователя прервалось.
- **When:** Соединение восстанавливается.
- **Then:** Клиент автоматически переподключается и получает актуальное состояние данных.

## 5. Нефункциональные требования

| Категория | Требование |
|-----------|------------|
| **Платформа** | Мобильный браузер: Safari на iOS, Chrome на Android, последние 2 мажорные версии. Десктоп поддерживается, но не является приоритетом. |
| **Целевое устройство** | iPhone 15: логический viewport 393 x 852 pt (1x CSS пикселей). Минимальный размер touch-target -- 44x44 pt согласно Apple HIG. |
| **Адаптивность** | Mobile-first вёрстка. Минимальная ширина viewport: 390px. |
| **Хранение данных** | SQLite через Django ORM. Сервер -- единственный источник правды. |
| **Реальное время** | WebSockets через Django Channels + Redis channel layer. Задержка трансляции изменений -- не более 1 секунды в локальной сети. |
| **Безопасность** | CSRF-защита включена. Пароли хранятся в хешированном виде (Django `AbstractUser`). Неавторизованный доступ к любому маршруту закрыт. |
| **Производительность** | Страница с 200 товарами в 10 категориях загружается менее чем за 2 секунды на локальном сервере. |
| **Локализация** | Только русский язык. |

## 7. Пользовательский путь и описание интерфейса

### 7.1 Карта экранов

```
/login           Страница входа
    |
    v (после авторизации)
/                Страница просмотра (главная)  <----+
    |  (переключатель в шапке)                      |
    v                                               |
/edit            Страница редактирования  ----------+

Примечание: /register отсутствует.
            Пользователей создаёт суперюзер:
              python manage.py createsuperuser
              -- или через /admin
```

### 7.2 Описание экранов

#### Экран: Вход (/login)

**Макет (сверху вниз):**
1. Название приложения по центру
2. Форма: поле «Email», поле «Пароль», кнопка «Войти»
3. Ссылки на регистрацию нет

**Поведение:**
- При ошибке авторизации -- inline-сообщение над формой: «Неверный email или пароль».
- Авторизованный пользователь, открывший `/login`, перенаправляется на `/`.

---

#### Экран: Просмотр (/)

**Макет (сверху вниз):**
1. **Шапка:**
   - Слева: переключатель «Просмотр | Редактирование» (активен «Просмотр»)
   - Справа: кнопка «Свернуть всё / Развернуть всё» + ссылка «Выйти»
2. **Список категорий** (отсортированы по приоритету, только с товарами к покупке):
   - Заголовок категории (нажимаемый, accordion)
   - Внутри: товары с `is_need_to_buy = True`, формат: `[чекбокс] Название -- количество ед.изм.`
   - Снятие галочки убирает товар из списка мгновенно

**Пустое состояние:** «Всё куплено! Список покупок пуст.»

---

#### Экран: Редактирование (/edit)

**Макет (сверху вниз):**
1. **Шапка:**
   - Слева: переключатель «Просмотр | Редактирование» (активен «Редактирование»)
   - Справа: кнопка «Свернуть всё / Развернуть всё» + ссылка «Выйти»

3. **Все категории** (отсортированы по приоритету):

### 8.4 Purchase

```python
class Purchase(models.Model):
    id             = models.AutoField(primary_key=True)
    name           = models.CharField(max_length=255)
    is_need_to_buy = models.BooleanField(default=True)
    quantity       = models.DecimalField(max_digits=8, decimal_places=2, default=1)
    category       = models.ForeignKey(Category, on_delete=models.CASCADE, related_name='purchases')
    unit           = models.ForeignKey(UnitOfMeasurement, on_delete=models.PROTECT, related_name='purchases')
    created_at     = models.DateTimeField(auto_now_add=True)
    updated_at     = models.DateTimeField(auto_now=True)

    class Meta:
        ordering = ['category__order', 'name']
        indexes  = [models.Index(fields=['is_need_to_buy'])]

    def __str__(self):
        return f"{self.name} ({self.quantity} {self.unit.abbreviation})"
```

### 8.5 Формат WebSocket-сообщений

> Консьюмер транслирует изменения в JSON. Клиент читает `type` и обновляет DOM.

```json
// Переключение чекбокса
{ "type": "purchase.updated", "purchase_id": 42, "is_need_to_buy": false }

// Создание категории
{ "type": "category.created", "category": { "id": 7, "name": "Молочное", "order": 2 } }
```

Поддерживаемые типы событий:
`purchase.created`, `purchase.updated`, `purchase.deleted`,
`category.created`, `category.updated`, `category.deleted`

## 9. Обработка ошибок и граничные случаи

| Ситуация | Ожидаемое поведение |
|----------|---------------------|
| БД недоступна при старте | Django вернёт 500. Показать кастомную страницу ошибки: «Сервис временно недоступен». |
| WebSocket-соединение разорвано | Тихий баннер: «Соединение потеряно, переподключение...». После восстановления -- баннер исчезает, данные обновляются. |
| Создание категории с пустым названием | Заблокировать сохранение. Inline-ошибка: «Введите название категории». |
| Создание товара с количеством <= 0 | Заблокировать сохранение. Inline-ошибка: «Количество должно быть больше 0». |
| Удаление категории с товарами | Диалог подтверждения: «Будут удалены N товаров. Продолжить?» |
| Два пользователя одновременно редактируют один объект | Last-write-wins: последнее сохранение перезаписывает более раннее без предупреждения. |
| Страница просмотра пуста | Пустое состояние: «Всё куплено! Список покупок пуст.» |
| Пользователь открывает несуществующий URL | Django вернёт 404 с кастомной страницей. |

```python
# tests/test_models.py
import pytest
from apps.shop.models import Category, Purchase, UnitOfMeasurement

@pytest.mark.django_db
def test_category_ordering_on_priority_conflict():
    """При создании категории с занятым приоритетом существующие сдвигаются вниз."""
    cat_a = Category.objects.create(name="Молочное", order=1)
    cat_b = Category.objects.create(name="Овощи", order=2)

    # Новая категория занимает приоритет 1 -- cat_a должна стать 2, cat_b -- 3
    Category.objects.create(name="Выпечка", order=1)

    cat_a.refresh_from_db()
    cat_b.refresh_from_db()
    assert cat_a.order == 2
    assert cat_b.order == 3


@pytest.mark.django_db
def test_purchase_disappears_from_view_when_unchecked(client, django_user_model):
    """Товар с is_need_to_buy=False не должен появляться на странице просмотра."""
    user = django_user_model.objects.create_user(username="tester", password="pass")
    client.force_login(user)

    unit = UnitOfMeasurement.objects.create(name="Штуки", abbreviation="шт.")
    cat  = Category.objects.create(name="Молочное", order=1)
    Purchase.objects.create(name="Молоко", category=cat, unit=unit, quantity=1, is_need_to_buy=False)

    response = client.get("/")
    assert "Молоко" not in response.content.decode()
```
